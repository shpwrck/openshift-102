= OpenShift 102 Workshop

Antora showroom site that bundles the Helm, advanced deployment, Istio, and Prometheus workshops maintained in separate repositories.

== Upstream workshops

* https://github.com/shpwrck/helm-workshop
* https://github.com/shpwrck/deploy-workshop
* https://github.com/shpwrck/istio-workshop
* https://github.com/shpwrck/prometheus-workshop

This repository vendors them as Git submodules under `workshops/` and exposes each as its own Antora component (see `content/bundled/*/antora.yml`) so they can coexist in a single generated site.

== Workshop CLI tools image

Releases also publish **`ghcr.io/<owner>/openshift-102-tools:<tag>`** (`Dockerfile.tools`): a UBI-based image learners run **on the cluster** so everyone shares the same CLIs without local installs. Most users paste **`deploy/openshift-102-tools-deployment.yaml`** (also in the offline tarball) into the OpenShift console **Import YAML** flow—no local `oc` required.

Included in that image:

* `oc` and `kubectl` (OpenShift stable client mirror)
* `helm` (pinned release) and `istioctl` (aligned with the Istio 1.23 workshop)
* `curl`, `wget`, `jq`, `git`, GnuPG (`gnupg2`), `skopeo`
* `/usr/local/bin/docker` — a **shim** only (no Docker Engine): `manifest inspect` uses `skopeo inspect --raw`; `pull` / `image inspect` / `history` use `skopeo` (and `jq` where needed)

Smoke checks: `bash scripts/verify-tools-image.sh IMAGE:TAG` (also run in CI on release). Offline layout and mirroring notes: `deploy/TOOLS.md`.

Interactive `bash` (for example from the pod **Terminal** tab after running `bash` or `bash -l`) sources `~/.bashrc`, which prints a one-time reminder of the bundled tools for this image.

== Clone and build

[source,bash]
----
git clone --recurse-submodules <repository-url>
cd openshift-102
./scripts/verify-submodules.sh
utilities/lab-build
utilities/lab-serve
----

Open http://localhost:8080/index.html .

The generated site substitutes default image coordinates for the **Shared CLI tools** page in the showroom (`tools-image-basename` / `tools-image-tag` in `default-site.yml`). Point them at your registry when building disconnected documentation, for example:

[source,bash]
----
antora generate default-site.yml --stacktrace \
  --attribute tools-image-basename=ghcr.io/myorg/openshift-102-tools \
  --attribute tools-image-tag=v1.0.4
----

If you already cloned without submodules:

[source,bash]
----
git submodule update --init --recursive
----

== GitHub Pages

The workflow in `.github/workflows/gh-pages.yml` publishes the `www/` output from `antora generate default-site.yml` using the same approach as the upstream workshop repositories.

Each run also packages the Helm chart under **`helm/`** on the same site and writes **`helm/index.yaml`**, so GitHub Pages doubles as a **live Helm chart repository** (no OCI needed for `helm repo add`).

One-time: turn Pages on for this repository and choose **GitHub Actions** as the build source.

* In the GitHub UI: **Settings** → **Pages** → **Build and deployment** → **Source** → **GitHub Actions**.
* Or from a machine with the https://cli.github.com/[`gh`] CLI: `gh auth login` then `./scripts/enable-github-pages.sh` (see `scripts/enable-github-pages.sh --help`).

With Pages enabled for your fork, replace `OWNER` and `REPO` with your GitHub `owner` and repository name (lowercase in the URL):

[source,bash]
----
helm repo add openshift-102-ghpages https://OWNER.github.io/REPO/helm
helm repo update
helm search repo openshift-102-ghpages
helm install my-workshop openshift-102-ghpages/openshift-102-workshop \
  --set image.tag=latest \
  --set image.repository=ghcr.io/OWNER/openshift-102
----

The chart **version** follows the Git **release tag** when the workflow runs on a **`v*`** tag (for example tag `v1.0.6` yields chart version **`1.0.6`**). Builds on **`main`** between releases use the SemVer from the latest **`v*`** tag on that commit plus **`+ci.<run number>`** (for example **`1.0.6+ci.4821`**) so each Pages deploy gets a distinct chart while staying aligned with the last release line. Set `image.repository` / `image.tag` to your GHCR showroom image (often the same `v*` as the chart when you cut a release).

== Disconnected OpenShift (air-gap)

* **Helm chart:** `deploy/helm/openshift-102-workshop` — deploys the pre-built static site using the UBI httpd runtime image (see `Dockerfile.runtime`).
* **SBOM:** CycloneDX JSON is produced in CI for the pushed runtime image (see `.github/workflows/release-offline.yml`).
* **Release:** Push a SemVer tag `v*` (for example `v1.0.4`) or run the **Release disconnected OpenShift bundle** workflow manually. Each release publishes the Helm package, an offline tarball (`www/` + chart + `DISCONNECTED.md` + `TOOLS.md` + `openshift-102-tools-deployment.yaml` + `images-mirror.txt`), CycloneDX SBOMs for the **showroom** and **CLI tools** images, and pushes `ghcr.io/<owner>/openshift-102:<tag>` and `ghcr.io/<owner>/openshift-102-tools:<tag>`.

Details: `deploy/DISCONNECTED.md`, `deploy/TOOLS.md`.
